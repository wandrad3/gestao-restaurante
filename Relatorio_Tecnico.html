1. Visão Geral

A aplicação fornece uma API REST para gerenciar tipos de usuário, usuários, restaurantes e itens de cardápio. O projeto utiliza separação entre domínio, aplicação e infraestrutura, persistência relacional com PostgreSQL, migrations Flyway, documentação OpenAPI, testes automatizados e execução integrada com Docker Compose.

Requisitos atendidos

  • CRUD de tipos de usuário, restaurantes e itens de cardápio.
  • Associação entre tipo de usuário, usuário, restaurante e cardápio.
  • Busca parcial e case-insensitive de usuários por nome.
  • Erros HTTP padronizados para validação, ausência e conflito.
  • Testes unitários e de integração com gate JaCoCo de 80%.
  • Dockerfile multi-stage e Docker Compose com healthcheck.
  • Collection e environment Postman.
  • Pipeline de integração contínua e abertura automática de PR.

3. Persistência e Infraestrutura

O PostgreSQL 16 foi escolhido por oferecer consistência ACID e suporte natural aos relacionamentos do domínio. O Flyway aplica a migration V1__create_initial_schema.sql. O Hibernate opera com ddl-auto=validate, evitando alterações implícitas no schema.

O Docker Compose fornece os serviços app e db. A aplicação aguarda o healthcheck do PostgreSQL, enquanto os dados são mantidos em volume persistente.

4. Testes e Qualidade

A suíte combina JUnit 5, Mockito, MockMvc, H2 e testes de integração. A busca por nome cobre nome exato, trecho parcial, diferença entre maiúsculas/minúsculas e ausência de resultado. O JaCoCo bloqueia o build quando a cobertura por instruções fica abaixo de 80%.

Comando de validação: ./mvnw clean verify -B

5. Pipeline de Integração Contínua

O workflow .github/workflows/ci.yml foi adaptado da Fase 1 para Java 17 e para o modelo atual. O objetivo é permitir que cada colaborador trabalhe em sua própria branch, faça push e receba uma PR automática somente após a validação da pipeline.

Branches monitoradas

  • feature/**: novas funcionalidades.
  • bugfix/**: correções de defeitos.
  • hotfix/**: correções urgentes.
  • chore/**: manutenção e infraestrutura.
  • refactor/**: refatorações.

Pull requests abertas para develop ou main também executam a validação. O evento workflow_dispatch permite execução manual.

Jobs

JobDependênciaResultado esperado
build-test Nenhuma Compila em Java 17, executa testes e valida cobertura >= 80%.
sonar build-test Executa SonarCloud quando secrets e variables estão configurados.
create-pr build-test e sonar Cria uma PR da branch do colaborador para develop.

Fluxo operacional

main
  |
  +-- develop
        |
        +-- feature/minha-funcionalidade
              |
              +-- push
                    |
                    +-- build + testes + JaCoCo
                    +-- SonarCloud (opcional)
                    +-- PR automática para develop
  1. O colaborador atualiza sua branch a partir de develop.
  2. Cria uma branch com um dos prefixos monitorados.
  3. Implementa, testa, faz commit e push.
  4. O GitHub Actions executa o workflow.
  5. Se os testes ou a cobertura falharem, nenhuma PR é criada.
  6. Se a pipeline passar, a PR para develop é criada.
  7. Novos pushes atualizam a mesma PR e reexecutam os checks.

Artefatos

  • Relatório JaCoCo mantido por 14 dias.
  • Relatórios Surefire mantidos por 7 dias.

6. Configuração Necessária no GitHub

O arquivo do workflow não consegue alterar sozinho as políticas do repositório. As configurações abaixo devem ser feitas por um administrador no GitHub.
  1. Em Settings > Actions > General > Workflow permissions, selecionar Read and write permissions.
  2. Marcar Allow GitHub Actions to create and approve pull requests.
  3. Criar ou confirmar a branch develop. O workflow tenta criá-la a partir de main quando ausente.
  4. Criar rulesets para develop e main, exigindo pull request e o check Build, tests and coverage.
  5. Desabilitar push direto em develop e main, exceto para administradores quando necessário.
  6. Importar separadamente .github/rulesets/develop.json e .github/rulesets/main.json em Settings > Rules > Rulesets. A interface não aceita um agregador contendo dois rulesets.
  7. Opcionalmente configurar o SonarCloud com o secret e as variables descritas acima.

8. Conclusão

A pipeline transforma o push em uma validação reproduzível e impede a abertura automática de PR quando compilação, testes ou cobertura falham. A separação entre branches de trabalho, develop e main estabelece um fluxo colaborativo com revisão antes da integração e da entrega.
